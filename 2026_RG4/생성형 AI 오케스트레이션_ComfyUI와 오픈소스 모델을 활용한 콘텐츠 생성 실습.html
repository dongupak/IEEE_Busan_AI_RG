1. ComfyUI 개요

  • 노드 기반 워크플로우 시스템: 시각적 프로그래밍 방식으로 생성형 AI 파이프라인 구축

  • 로컬 실행: 개인 하드웨어에서 완전한 제어권 확보

2. 수치 정밀도 (Numerical Precision)

2.1 기본 개념

수치 정밀도는 모델의 가중치와 활성화 값을 표현하는 비트 수를 의미합니다. 정밀도가 높을수록 정확도는 높아지지만, 메모리 사용량과 계산 비용이 증가합니다.

2.2 부동소수점 형식 비교

FP64 (Double Precision Floating Point)

  • 비트 구성: 64비트
  • 정밀도: 매우 높음 (약 15-17 자릿수)
  • 용도: 과학 계산, 정밀한 수치 시뮬레이션
  • AI 사용: 거의 사용되지 않음 (오버킬)

FP32 (Single Precision Floating Point)

  • 비트 구성: 32비트 (부호 1비트 + 지수 8비트 + 가수 23비트)
  • 정밀도: 높음 (약 6-9 자릿수)
  • 동적 범위: ±3.4 × 10³⁸
  • 용도: 일반적인 딥러닝 학습 및 추론
  • 메모리: 모델 파라미터당 4바이트
  • 특징: 안정성과 정확도의 균형

FP16 (Half Precision Floating Point / IEEE 754 Half Precision)

  • 비트 구성: 16비트 (부호 1비트 + 지수 5비트 + 가수 10비트)
  • 정밀도: 중간 (약 3-4 자릿수)
  • 동적 범위: ±65,504 (매우 제한적)
  • 장점:
    • FP32 대비 50% 메모리 절감
    • 더 높은 수치 정밀도 (2¹⁰ ≈ 1024 단계)
    • 추론 시 세밀한 표현 가능
  • 단점:
    • 학습 시 그래디언트 언더플로우/오버플로우 위험
    • ±65,000 범위 초과 시 스케일링 필요
  • 용도: 추론, 혼합 정밀도 학습
  • 메모리: 모델 파라미터당 2바이트

BF16 (Brain Floating Point 16)

  • 비트 구성: 16비트 (부호 1비트 + 지수 8비트 + 가수 7비트)
  • 정밀도: 낮음 (약 2-3 자릿수)
  • 동적 범위: FP32와 동일 (±3.4 × 10³⁸)
  • 장점:
    • FP32와 같은 지수 범위 → 수치 안정성 우수
    • 그래디언트 스케일링 불필요
    • 학습 시 더 안정적
    • 더 작은 수 표현 가능 (9.18e-41까지)
  • 단점:
    • FP16보다 낮은 정밀도 (2⁷ ≈ 128 단계)
    • 0에 가까운 값은 0으로 반올림될 위험
  • 용도: 대규모 언어 모델 학습 (TPU, 최신 GPU)
  • 메모리: 모델 파라미터당 2바이트
  • 주요 사용: Google TPU, NVIDIA A100 이상

TF32 (TensorFloat-32)

  • 비트 구성: 내부적으로 19비트 (부호 1비트 + 지수 8비트 + 가수 10비트)
  • 특징:
    • NVIDIA Ampere 아키텍처(A100, RTX 3090 이상) 전용
    • Tensor Core 연산에서 자동 사용
    • FP32 코드를 수정하지 않고 가속
    • 저장은 FP32, 연산은 TF32
  • 용도: 딥러닝 학습 가속

FP8 (8-bit Floating Point)

  • 비트 구성: 8비트 (여러 변형 존재)
    • E4M3: 지수 4비트 + 가수 3비트
    • E5M2: 지수 5비트 + 가수 2비트
  • 장점:
    • FP16 대비 50% 추가 메모리 절감
    • 최신 GPU(H100, H200)에서 하드웨어 가속
  • 단점: 매우 낮은 정밀도
  • 용도: 대규모 모델 추론, 최첨단 학습

INT8 (8-bit Integer)

  • 비트 구성: 8비트 정수
  • 범위: -128 ~ 127 (signed) 또는 0 ~ 255 (unsigned)
  • 용도: 양자화된 모델 추론
  • 장점: 매우 빠른 연산 속도, 적은 메모리
  • 단점: 정밀도 손실

2.5 실무 선택 가이드

학습 (Training):

  • 일반적: FP32 또는 혼합 정밀도 (FP32 + FP16/BF16)
  • 대규모 LLM: BF16 (안정성 우수)
  • 최신 GPU (A100+): TF32 자동 활성화

추론 (Inference):

  • 품질 우선: FP16 (24배 더 높은 수치 정밀도 vs BF16)
  • 메모리 제약: INT8 양자화 또는 FP8
  • 균형: BF16 또는 FP16

ComfyUI에서:

  • SDXL/SD3: BF16 또는 FP16
  • Flux: FP16 권장 (프롬프트 이해도 향상)
  • T5 인코더: FP8 또는 FP16 (메모리 vs 품질 트레이드오프)

3. 모델 파일 형식

3.1 Safetensors (.safetensors)

개요

  • 개발: Hugging Face
  • 형식: 바이너리 파일 형식
  • 보안: 임의 코드 실행 방지 (pickle 취약점 해결)
  • 속도: 빠른 로딩 및 저장
  • 메모리 안전: 메모리 매핑(mmap) 지원

구조

┌─────────────────────┐
│   메타데이터 헤더    │  ← JSON 형식, 모델 정보
├─────────────────────┤
│   텐서 메타데이터    │  ← 텐서 이름, 크기, dtype, 오프셋
├─────────────────────┤
│     텐서 데이터      │  ← 실제 가중치 데이터
└─────────────────────┘

장점

  • ✅ 보안: .ckpt 파일의 pickle 취약점 없음
  • ✅ 빠른 로딩: 메모리 매핑으로 전체 파일 로드 불필요
  • ✅ 크로스 플랫폼: Python, Rust, JavaScript에서 사용 가능
  • ✅ 호환성: PyTorch, TensorFlow, JAX 지원

단점

  • ❌ Python 객체 직렬화 불가 (오직 텐서만)

파일명 예시

sd_xl_base_1.0.safetensors           # SDXL 기본 모델
flux1-dev-fp16.safetensors           # Flux Dev FP16
sdxl_vae.safetensors                 # SDXL VAE
clip_l.safetensors                   # CLIP-L 텍스트 인코더
t5xxl_fp8_e4m3fn.safetensors         # T5-XXL FP8 인코더

3.2 Checkpoint (.ckpt, .pt, .pth)

개요

  • 형식: PyTorch pickle 기반
  • 역사적 표준: 이전 Stable Diffusion 모델에 널리 사용
  • 보안 우려: 임의 코드 실행 가능

장점

  • ✅ 복잡한 Python 객체 저장 가능
  • ✅ 레거시 호환성

단점

  • ❌ 보안 위험: 악성 코드 포함 가능
  • ❌ 느린 로딩 속도
  • ❌ 큰 파일 크기

권장사항

⚠️ ComfyUI에서는 .safetensors 사용을 강력히 권장

3.3 GGUF (GPT-Generated Unified Format)

개요

  • 개발: llama.cpp 프로젝트 (Georgi Gerganov)
  • 목적: 빠른 로딩과 효율적인 추론을 위한 바이너리 형식
  • 주요 사용: 대규모 언어 모델 (LLM) 양자화 및 배포
  • 후속 형식: GGML, GGMF, GGJT의 개선판

구조

┌──────────────────────────┐
│   메타데이터 섹션         │  ← 키-밸류 쌍 (모델 아키텍처, 버전, 하이퍼파라미터)
├──────────────────────────┤
│   텐서 메타데이터 섹션    │  ← 텐서 이름, 크기, dtype, 오프셋
├──────────────────────────┤
│   텐서 데이터 섹션        │  ← 양자화된 가중치 데이터
└──────────────────────────┘

양자화 스킴 (Quantization Schemes)

GGUF의 핵심 특징은 유연한 양자화 지원입니다:

  • Q4_K_M: 4비트, K-평균 클러스터링, 중간 블록 크기
  • Q5_K_M: 5비트, K-평균 클러스터링, 중간 블록 크기
  • Q8_0: 8비트, 기본 선형 양자화
  • IQ4_XS: 4비트, 초소형 블록 (고급 압축)

장점

  • ✅ CPU 최적화: 소비자 하드웨어에서 효율적 실행
  • ✅ 빠른 로딩: mmap 지원으로 초고속 모델 로드
  • ✅ 단일 파일: 모든 정보를 하나의 파일에 포함
  • ✅ 확장성: 메타데이터 추가 시 하위 호환성 유지
  • ✅ 양자화: 다양한 정밀도 옵션 (Q2~Q8)

단점

  • ❌ 제한적 사용: 주로 LLM, 디퓨전 모델은 덜 일반적
  • ❌ GPU 가속 제한: CPU 추론에 최적화

지원 프레임워크

  • llama.cpp
  • Ollama
  • vLLM
  • GPT4All
  • LM Studio

사용 예시

# Hugging Face 모델을 GGUF로 변환
python convert-hf-to-gguf.py --model-id meta-llama/Llama-3-8B

# Ollama에서 GGUF 모델 실행
ollama run llama3:8b-q4_k_m

GGUF 파일명 예시

mistral-7b-instruct-q5_k_m.gguf      # Mistral 7B, Q5_K_M 양자화
phi-3-mini-q8_0.gguf                 # Phi-3 Mini, Q8_0 양자화

4. 텍스트 인코더 (Text Encoders)

4.1 텍스트 인코더의 역할

텍스트 인코더는 자연어 프롬프트를 디퓨전 모델이 이해할 수 있는 수치 벡터(임베딩)로 변환합니다.

"A cat sitting on a mat" → [0.23, -0.45, 0.89, ...] (임베딩 벡터)
                              ↓
                        디퓨전 모델 조건화
                              ↓
                          이미지 생성

4.2 CLIP (Contrastive Language-Image Pre-training)

개념

  • 개발: OpenAI (2021)
  • 학습 방법: 대조 학습 (이미지-텍스트 쌍 4억 개)
  • 목적: 텍스트와 이미지를 공유 임베딩 공간에 매핑

CLIP 변형 비교

CLIP-L (Large)
  • 모델 크기: ViT-L/14 (Large Vision Transformer)
  • 파라미터: 약 427M (4억 2700만)
  • 임베딩 차원: 768차원
  • 토큰 제한: 77 토큰 (실효 길이 ~20 토큰)
  • 특징:
    • 키워드 중심 이해
    • 간단한 프롬프트에 효과적
    • 낮은 메모리 사용량
  • 사용 예: Stable Diffusion 1.x, 2.x
CLIP-G (Giant)
  • 모델 크기: ViT-G/14 (Giant Vision Transformer)
  • 파라미터: 약 1.4B (14억)
  • 임베딩 차원: 1280차원
  • 토큰 제한: 77 토큰
  • 특징:
    • 더 균형잡힌 데이터 분포
    • 복잡한 프롬프트 이해 향상
    • CLIP-L보다 높은 표현력
  • 사용 예: Stable Diffusion XL (SDXL)
CLIP-H (Huge)
  • 파라미터: 약 632M (6억 3200만)
  • 임베딩 차원: 1024차원
  • 특징: CLIP-L과 CLIP-G 사이의 중간 크기
  • 사용 예: 일부 실험적 모델

CLIP 한계

⚠️ 77 토큰 제한: 긴 상세한 프롬프트 처리 어려움 ⚠️ 실효 길이: 20 토큰 이후 성능 저하 ⚠️ 문법 이해 부족: 키워드 중심, 문장 구조 이해 약함

Long-CLIP

  • 개선점: 248 토큰까지 지원
  • 방법:
    • Knowledge-Preserved Stretching (위치 임베딩 확장)
    • Primary Component Matching (CLIP 특성 유지)
  • 성능: 긴 캡션 검색 25% 향상
  • 호환성: 기존 CLIP 교체 가능 (Stable Diffusion 플러그인)

4.3 T5 (Text-To-Text Transfer Transformer)

개념

  • 개발: Google Research (2019)

T5 변형 비교

T5-Small
  • 파라미터: 60M (6천만)
  • 레이어: 인코더 6층, 디코더 6층
  • 임베딩 차원: 512
  • 용도: 경량 실험
T5-Base

  • 레이어: 인코더 12층, 디코더 12층
  • 임베딩 차원: 768
  • 특징:

  • 메모리: 매우 효율적
T5-Large
  • 파라미터: 770M (7억 7천만)
  • 레이어: 인코더 24층, 디코더 24층
  • 임베딩 차원: 1024
T5-XL
  • 파라미터: 3B (30억)
  • 레이어: 인코더 24층, 디코더 24층
  • 임베딩 차원: 2048
  • 특징:
    • T5-XXL 성능의 97% 유지
    • 4배 작은 크기
    • 실용적 선택
T5-XXL (Extra Extra Large)
  • Full name: Text-To-Text Transfer Transformer - Extra Extra Large
  • 파라미터: 11B (110억)
  • 레이어: 인코더 24층, 디코더 24층
  • 임베딩 차원: 4096
  • 피드포워드 차원: 65,536
  • 어텐션 헤드: 128
  • 특징:
    • 복잡한 프롬프트 이해 우수
    • 긴 문장 구조 파악
    • 텍스트 렌더링 (이미지 내 글자 생성) 능력
    • 공간 관계 이해 (2D, 3D)
    • 숫자 개념 (numeracy)
  • 단점:

    • Google Imagen

5. VAE (Variational Autoencoder)

5.1 VAE의 역할

개념

VAE는 이미지와 잠재 공간(latent space) 사이를 변환하는 신경망입니다. 디퓨전 모델은 고해상도 픽셀 공간이 아닌 압축된 잠재 공간에서 작동합니다.

구조

이미지 (512×512×3)
    ↓ Encoder
잠재 표현 (64×64×4) ← 디퓨전 모델이 여기서 작동
    ↓ Decoder
재구성 이미지 (512×512×3)

5.2 VAE 컴포넌트

VAE Encoder

  • 입력: 픽셀 이미지 (RGB, 예: 512×512×3)
  • 출력: 잠재 표현 (예: 64×64×4)
  • 압축 비율: 일반적으로 8배 (8×8×1 픽셀 → 1×1×4 잠재)
  • 역할:
    • 이미지를 압축된 벡터 공간으로 인코딩
    • 노이즈 추가 및 변환 준비
  • 사용 시점: 이미지→잠재 변환 (Image-to-Image, Inpainting)

VAE Decoder

  • 입력: 잠재 표현 (예: 64×64×4)
  • 출력: 픽셀 이미지 (RGB, 예: 512×512×3)
  • 역할:
    • 잠재 공간의 압축된 데이터를 실제 이미지로 복원
    • 최종 생성 이미지 출력
  • 사용 시점: 생성 파이프라인의 마지막 단계

5.3 주요 VAE 모델

SD1.5 VAE

  • 적용 모델: Stable Diffusion 1.x, 2.x
  • 잠재 차원: 4 채널
  • 압축 비율: 8x
  • 특징: 기본 VAE, 가끔 색상 왜곡 발생

SDXL VAE

  • 파일명: sdxl_vae.safetensors
  • 적용 모델: Stable Diffusion XL (SDXL) 전용
  • 특징:
    • SDXL 잠재 분포에 최적화
    • SD1.5 VAE와 호환 불가
    • 더 나은 색상 정확도
  • VRAM: 약 2.5GB
  • 주의사항:
    • ⚠️ 파일명 변경 금지 (.safetensors 확장자 유지)
    • ⚠️ 다중 VAE 로드 금지 (파이프라인 혼란)

SD3 VAE

  • 적용 모델: Stable Diffusion 3
  • 개선사항: 향상된 오토인코딩 모델
  • 특징: 더 나은 디테일 보존

원인: 파일명 또는 확장자 변경 해결:

  • 원본 파일명 복원
  • .safetensors 확장자 확인
  • models/vae/ 폴더에 위치 확인

문제 2: 색상 왜곡 또는 artifact

원인: 잘못된 VAE 사용 (모델 불일치) 해결:

  • SD1.5/2.x 모델 → 해당 VAE 사용

원인: VAE 디코딩 시 높은 VRAM 사용 해결:

  • 해상도 낮추기
  • Tiled VAE 사용 (큰 이미지 분할 처리)

6. 양자화 (Quantization)

6.1 양자화란?

개념

양자화는 모델의 가중치와 활성화 값을 낮은 비트 정밀도로 변환하여 메모리 사용량과 연산량을 줄이는 압축 기법입니다.

예시

원본 FP32: 3.14159265359 (32비트)
    ↓ 양자화
INT8:      3 (8비트, 4배 압축)

6.2 양자화 스킴 (GGUF 기반)

Q2_K (2-bit K-quantization)

  • 메모리 절감: FP16 대비 ~75%
  • 품질 저하: 매우 심각 (30%+ 성능 손실)
  • 속도: 4.0-5.0배 빠름
  • 용도: 실험적, 극한 리소스 제약
  • 권장: ❌ 실용적이지 않음

Q3_K_M (3-bit K-quantization, Medium)

  • 메모리 절감: FP16 대비 ~70%
  • 품질 저하: 심각 (15-30% 성능 손실)
  • 속도: 3.5-4.5배 빠름
  • 용도: 매우 제한된 하드웨어
  • 권장: △ 소비자 하드웨어 최소 요구사항

Q4_K_M (4-bit K-quantization, Medium)

  • Full name: 4-bit K-means Quantization - Medium Block Size
  • 메모리 절감: FP16 대비 ~50%
  • 품질 저하: 낮음 (5-15% 성능 손실, 대부분 모델에서 95% 품질 유지)
  • 속도: 3.0-4.0배 빠름
  • 방법:
    • 4비트 정밀도
    • K-평균 클러스터링으로 유사 가중치 그룹화
    • 중간 크기 블록 (밀집 레이어 최소 왜곡)
  • 용도: 가장 널리 사용되는 선택 (소비자 하드웨어 배포)
  • 권장: ✅✅ 대부분의 사용 사례에 최적

실용 예시:

Llama 2 13B:
- FP16: 32GB RAM 필요 (불가능 on 16GB PC)
- Q4_K_M: 12GB RAM 필요 (가능) + 2배 속도 향상 + 95% 품질

Q4_K_S (4-bit K-quantization, Small)

  • 메모리 절감: Q4_K_M과 유사
  • 품질 저하: Q4_K_M보다 약간 높음
  • 블록 크기: 더 작음 (더 공격적인 압축)
  • 용도: 메모리 극한 최적화
  • 권장: △ Q4_K_M 선호

Q5_K_M (5-bit K-quantization, Medium)

  • 메모리 절감: FP16 대비 ~40%
  • 품질 저하: 최소 (5% 미만 성능 손실)
  • 속도: 2.5-3.2배 빠름
  • 용도: 높은 품질이 필요한 애플리케이션
  • 권장: ✅ 품질 우선 시 선택

Q6_K (6-bit K-quantization)

  • 메모리 절감: FP16 대비 ~30%
  • 품질 저하: 매우 낮음 (3% 미만)
  • 속도: 2.2-2.8배 빠름
  • 용도: 거의 원본 품질 요구
  • 권장: ✅ 높은 품질 + 적당한 압축

Q8_0 (8-bit Quantization)

  • Full name: 8-bit Linear Quantization (Type 0)
  • 메모리 절감: FP16 대비 ~20%
  • 품질 저하: 거의 없음 (2% 미만)
  • 속도: 2.0-2.5배 빠름
  • 방법: 기본 선형(균일) 양자화, 고급 기법 없음 (0)
  • 용도: 프로덕션 애플리케이션 (높은 품질 필수)
  • 특징: BF16 네이티브 모델도 품질 잘 유지
  • 권장: ✅ 연구 및 개발, 고품질 추론

Stable Diffusion 1.x/2.x

  • Full name: Multimodal Diffusion Transformer
  • 파라미터 범위: 800M ~ 8B (여러 변형)
  • 해상도: 1024×1024 이상
  • 텍스트 인코더: CLIP-L + CLIP-G + T5-XXL

주요 개선사항:

    • 이미지와 언어에 별도의 가중치 세트 사용
    • 두 모달리티가 독립적으로 작동하면서 상호 참조
    • 트랜스포머 기반 (U-Net 대체)

  2. Rectified Flow (RF):

    • 데이터와 노이즈를 선형 경로로 연결
    • 확률적 샘플링 → 결정론적 벡터 필드
    • 더 적은 샘플링 스텝으로 고품질 생성
    • 재가중 RF: 경로 중간 부분에 더 많은 가중치

  3. 유연한 텍스트 인코더:

    • T5-XXL 제거 가능 (메모리 절감)
    • 시각적 미학 유지 (win rate 50%)
    • 타이포그래피 성능 약간 저하 (win rate 38%)

성능:

  • 8B 모델: RTX 4090 (24GB VRAM)에서 34초 (1024×1024, 50 스텝)
  • DALL·E 3, Midjourney v6, Ideogram v1 능가

장점:

  • ✅ 최고 수준의 텍스트 이해
  • ✅ 정확한 철자 (텍스트 렌더링)
  • ✅ 복잡한 프롬프트 준수
  • ✅ 확장성 (여러 크기 모델)

단점:

  • ❌ 높은 VRAM 요구량
  • ❌ 느린 생성 속도

Double-Stream Blocks:

Single-Stream Blocks:

  • 가중치: 공유
  • 특징:
    • 경량화
    • 효율적 용량 확장
    • Flux 고유 혁신

3. Flow Matching

  • 방식: 결정론적 ODE (Ordinary Differential Equation)
  • 샘플러: FlowMatchEulerDiscreteScheduler
  • 장점:
    • 확률적 노이즈 스케줄 불필요
    • 더 빠르고 안정적인 생성
    • 복잡도 감소

4. Rotary Positional Embeddings (RoPE)

  • 목적: 위치 정보 인코딩
  • 장점: 더 나은 공간 관계 이해

5. 병렬 어텐션 레이어

  • 동시 처리: 여러 어텐션 헤드 병렬 실행
  • 효율성: 속도 향상

2. Latent Flow Matching Architecture

기존 디퓨전 vs Flow Matching:

기존 디퓨전 모델:
노이즈 → 반복적 디노이징 → 이미지 (많은 스텝 필요)

Flux 2 Flow Matching:
텍스트 설명 → 직접 매핑 → 잠재 공간 → 이미지 (적은 스텝)

장점:

  • ✅ 빠른 수렴: 고품질 출력을 위한 샘플링 스텝 감소
  • ✅ 향상된 프롬프트 준수: 복잡한 지시사항 정확한 해석
  • ✅ 일관성 개선: 시드 제어로 재현 가능한 결과

8. LoRA와 파인튜닝

8.1 LoRA 개념

Full Name

LoRA: Low-Rank Adaptation

개념

LoRA는 대규모 사전학습 모델을 효율적으로 파인튜닝하는 기법입니다. 전체 모델 가중치를 업데이트하는 대신, 저랭크(low-rank) 행렬 분해를 사용하여 훨씬 적은 파라미터만 학습합니다.

수식

W' = W₀ + ΔW = W₀ + β · B · A

여기서:
- W₀: 원본 고정 가중치 (frozen)
- ΔW: 학습 가능한 업데이트 (B·A 저랭크 분해)
- A: 랭크 r의 다운 프로젝션 행렬 (d × r)
- B: 랭크 r의 업 프로젝션 행렬 (r × d)
- β: 스케일링 팩터 (보통 α/r, α는 하이퍼파라미터)
- r: 랭크 (일반적으로 4, 8, 16, 32, 64)

예시

원본 가중치: 1024 × 1024 = 1,048,576 파라미터
LoRA (rank=8):
  - A: 1024 × 8 = 8,192 파라미터
  - B: 8 × 1024 = 8,192 파라미터
  - 총: 16,384 파라미터 (약 1/64로 감소)

8.2 LoRA 장점

1. 메모리 효율성

  • 전체 파인튜닝 대비 수백 배 적은 파라미터
  • 소비자 GPU에서 파인튜닝 가능

2. 빠른 학습

  • 더 적은 파라미터 → 더 빠른 수렴
  • 적은 데이터로도 효과적

3. 모듈성

  • 여러 LoRA 어댑터 생성 가능
  • 베이스 모델에 동적으로 병합

4. 저장 공간

  • LoRA 파일: 일반적으로 10-200MB
  • 전체 모델: 수 GB
  • 여러 스타일/개념을 작은 파일로 저장

5. 조합 가능

  • 여러 LoRA를 동시에 적용 가능
  • 가중치 조절로 영향력 제어

8.3 LoRA 사용 사례

스타일 학습

  • 특정 예술 스타일 (애니메이션, 유화, 수채화 등)
  • 작가의 독특한 화풍

캐릭터/객체 학습

  • 특정 인물, 캐릭터, 마스코트
  • 제품, 로고, 브랜딩 요소

개념 학습

  • 추상적 개념 (예: "드림코어", "사이버펑크")
  • 특정 조명, 분위기

도메인 특화

  • 의료 영상
  • 건축 렌더링
  • 패션 디자인

8.4 LoRA 파인튜닝 프로세스

1. 데이터 준비

  • 이미지 수집: 5-50장 (스타일) 또는 15-100장 (캐릭터)
  • 캡셔닝: 각 이미지에 상세한 텍스트 설명
  • 전처리: 크기 조정, 크롭, 정규화

2. 학습 설정

  • 베이스 모델: SD1.5, SDXL, Flux 등
  • 랭크 (r): 4-64 (높을수록 표현력 증가, 메모리 증가)
  • Alpha (α): 보통 랭크와 동일
  • 학습률: 1e-4 ~ 1e-5
  • 학습 스텝: 500-5000

3. 학습 실행

  • 방법: DreamBooth, Textual Inversion, 또는 일반 LoRA
  • 정규화: Prior preservation loss (선택사항)
  • 모니터링: 과적합 방지

4. LoRA 저장

  • 파일 형식: .safetensors (권장) 또는 .pt
  • 크기: 일반적으로 10-200MB
  • 위치: models/loras/

8.5 LoRA 병합

가중치 합산 방식

W_merged = W₀ + β₁·B₁·A₁ + β₂·B₂·A₂ + ...

예시:
- 베이스 모델 (100%)
- + LoRA 1 (스타일) × 0.7
- + LoRA 2 (캐릭터) × 1.0
- + LoRA 3 (조명) × 0.5

ComfyUI에서 LoRA 사용

  1. Load LoRA 노드 추가
  2. LoRA 파일 선택
  3. 강도 조절: 0.0-2.0 (일반적으로 0.7-1.0)
  4. 모델 및 CLIP에 연결

8.6 LoRA 변형

AdaLoRA

  • Full name: Adaptive Low-Rank Adaptation
  • 특징: 학습 중 랭크 동적 조절
  • 장점: 더 효율적인 파라미터 사용

LoHa (Locon)

  • Full name: Low-Rank Hadamard Adaptation
  • 방법: Hadamard product 사용
  • 특징: 더 복잡한 변환 가능

LyCORIS

  • Full name: Lora beYond Conventional methods, Other Rank adaptation Implementations for Stable diffusion
  • 특징: LoRA의 일반화된 형태

9.2 KSampler 파라미터 이해

Steps (샘플링 스텝)

  • SD1.5/2.x: 20-50 스텝
  • SDXL: 20-40 스텝
  • SD3: 20-30 스텝
  • Flux Dev: 20-50 스텝
  • Flux Schnell: 3-6 스텝 (초고속)
  • 원칙: 더 많은 스텝 = 더 정제된 이미지, 더 긴 시간

CFG Scale (Classifier-Free Guidance)

  • 범위: 1.0-20.0
  • 일반적: 7.0-8.0
  • 낮음 (1-5): 창의적, 덜 제약적, 프롬프트 이탈 가능
  • 중간 (7-10): 균형잡힌 준수도
  • 높음 (15-20): 엄격한 프롬프트 준수, 과포화 위험
  • 특수: Flux Schnell, SD3 Turbo는 0으로 설정

Sampler (샘플러)

  • Euler a: 빠르고 창의적
  • DPM++ 2M Karras: 고품질, 안정적 (권장)
  • DPM++ SDE Karras: 디테일 높음, 느림
  • DDIM: 재현 가능, 결정론적
  • UniPC: 매우 빠름
  • FlowMatchEulerDiscrete: Flux 전용

Scheduler (스케줄러)

  • Karras: 일반적 추천
  • Exponential: 부드러운 전환
  • Simple: 기본

Denoise (노이즈 제거 강도)

  • 1.0: 완전히 새로운 이미지 생성 (Text-to-Image)
  • 0.5-0.8: 원본 유지하면서 변형 (Image-to-Image)
  • 0.3-0.5: 약간의 수정
  • 0.1-0.3: 매우 미세한 조정

9.5 VRAM 최적화 전략

  2. 해상도 낮추기: 512×512 또는 768×768
  3. 배치 크기: 1로 설정
  4. VAE Tiling: 큰 이미지 분할 처리
  5. 텍스트 인코더 오프로딩: T5-XXL 제외 또는 FP8 사용

중급 GPU (10-12GB VRAM)

  1. 모델 선택: SDXL, SD3 (작은 변형)
  2. 해상도: 1024×1024 가능
  3. LoRA: 여러 개 동시 사용 가능

고급 GPU (16-24GB VRAM)

  1. 모델 선택: SD3, Flux Dev, 모든 모델
  2. 해상도: 1024×1024 이상
  3. 배치 생성: 동시에 여러 이미지 가능

9.6 품질 향상 기법

1. Hires Fix (고해상도 수정)

  • 방법: 낮은 해상도로 생성 후 업스케일 + 재샘플링
  • 효과: 디테일 향상, artifact 감소
  • 설정: 업스케일 2x, denoise 0.4-0.6

2. ControlNet

  • 목적: 포즈, 깊이, 엣지 등으로 제어
  • 종류:
    • Canny: 엣지 검출
    • Depth: 깊이 맵
    • OpenPose: 포즈 제어
    • Scribble: 스케치 기반

3. Inpainting (인페인팅)

  • 목적: 이미지 일부 수정
  • 프로세스:
    1. 마스크 생성 (수정할 영역)
    2. 프롬프트 작성
    3. Inpainting 모델 사용
    4. denoise 0.6-1.0

4. Img2Img Refinement

  • 목적: 기존 이미지 개선
  • 설정: denoise 0.3-0.5

9.8 트러블슈팅

문제 1: 검은 이미지 생성

원인: VAE 오류, VRAM 부족 해결:

  • 다른 VAE 시도
  • 해상도 낮추기
  • FP16 → FP32 VAE

문제 2: 프롬프트가 무시됨

원인: CFG Scale 너무 낮음, 잘못된 인코더 해결:

  • CFG Scale 7-8로 설정
  • 텍스트 인코더 확인 (CLIP, T5 로드 확인)

문제 3: 과포화/이상한 색상

원인: CFG Scale 너무 높음 해결:

  • CFG Scale 7-10으로 낮추기

문제 4: Out of Memory (OOM)

원인: VRAM 초과 해결:

  • 해상도 낮추기
  • 배치 크기 1로
  • FP16 모델 사용
  • 텍스트 인코더 오프로딩

문제 5: 느린 생성 속도

원인: CPU 병목, 비효율적 샘플러 해결:

  • DPM++ 2M Karras 샘플러 사용
  • 스텝 수 감소
  • Flux Schnell 사용 (3-6 스텝)
